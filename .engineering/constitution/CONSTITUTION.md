# CONSTITUTION.md – v1

*This constitution defines the shared engineering philosophy, governance, and best‑practice standards for all contributors—human and AI—working in the **autonomous-coconut-harvester** repository.*

---

## 1. Purpose & Scope
- Establish a **common decision‑making framework** that is independent of any specific feature implementation.
- Provide **lightweight, enforceable rules** that keep the codebase healthy, secure, and maintainable.
- Serve as the **single source of truth** for engineering culture, tooling conventions, and review processes.
- Apply to **all code, documentation, workflow scripts, and infrastructure** under the repository root.

---

## 2. Core Principles
| Principle | Guideline |
|-----------|-----------|
| **Simplicity** | Prefer the simplest solution that satisfies requirements. Avoid premature optimisation and unnecessary abstraction. |
| **Transparency** | All decisions, changes, and rationale must be recorded (e.g., pull‑request description, ADR, or comment). |
| **Automation First** | Repetitive tasks (linting, testing, CI, documentation generation) are automated and run on every PR. |
| **Safety** | Code changes must not introduce regressions. Require automated tests and, where applicable, integration verification before merge. |
| **Modularity** | Keep components loosely coupled. Public interfaces should be stable; internal implementation may evolve freely. |
| **Traceability** | Every public API, database schema, and workflow artefact must have a corresponding spec entry. |
| **Collaboration** | Human reviewers and AI agents share the same standards. AI‑generated changes are subject to the same review checklist. |
| **Continuous Improvement** | Periodic retrospectives evaluate whether the constitution itself needs amendment. |

---

## 3. Governance
1. **Stewardship** – A rotating **Engineering Steward** (human) has final authority on exceptional cases (e.g., breaking change, security emergency) and maintains the constitution.
2. **Decision Process** – For any change that:
   - Alters a public contract (API, DB schema, CLI), or
   - Introduces a new dependency, or
   - Impacts security/performance
   
   The proposer must create an **Architecture Decision Record (ADR)** in `knowledge/` and obtain at least **one affirmative review** from a qualified reviewer.
3. **Review Ownership** – Every pull request must have an **owner** (the author) and at least one **independent reviewer** (human or vetted AI) who signs off.
4. **Conflict Resolution** – If reviewers disagree, the issue escalates to the current Engineering Steward for a short‑term decision.

---

## 4. Code Quality Standards
- **Linting** – Run `npm run lint` (frontend) and `ruff`/`flake8` (backend) on CI. Lint failures block merges.
- **Testing** – Minimum 80 % coverage for new code. All tests must pass locally before CI.
- **Type Safety** – TypeScript types must be strict (`noImplicitAny`). Python code uses type hints and `mypy` in CI.
- **Documentation** – Public functions/classes must have docstrings/comments; high‑level design documented in `knowledge/`.
- **Security** – No hard‑coded secrets. Dependencies scanned via `dependabot`/`snyk`. Critical vulnerabilities must be patched within 48 h.
- **Performance** – Benchmarks for latency‑critical paths must be included in `tests/performance/` and must not regress by >5 %.

---

## 5. Workflow Conventions
- **Branch Model** – Feature branches (`feat/<name>`), bugfix branches (`bug/<name>`), and short‑lived `dev` branches. All merges go through PRs targeting `main`.
- **Commit Messages** – Follow Conventional Commits (`type(scope): description`). Example: `feat(robot): add path‑planning algorithm`.
- **CI Pipeline** – Lint → Unit Tests → Integration Tests → Security Scan → Deploy (if applicable).
- **AI‑Generated Changes** – Must be clearly marked in PR description (`[AI]`). The same review checklist applies.
- **Change Log** – `CHANGELOG.md` is autogenerated from PR titles on merge.

---

## 6. Tooling & Infrastructure
- **File Organization** – All engineering artifacts live under the `.engineering/` directory (prompts, specs, templates, workflows, reviews). This keeps the production code clean.
- **Knowledge Base** – Use the `knowledge/` directory for ADRs, design diagrams, and external references.
- **Development Environment** – Standardised via `devcontainer.json` (if present) and documented in `README.md`.
- **Versioning** – Semantic versioning (`MAJOR.MINOR.PATCH`) for released artefacts; internal packages use `pyproject.toml`/`package.json` version fields.

---

## 7. Review Checklist (Summary)
1. **Scope** – Does the PR stay within the stated scope?
2. **Tests** – Are there unit/integration tests for new logic?
3. **Docs** – Are public interfaces documented?
4. **Security** – No secrets, dependencies up‑to‑date?
5. **Performance** – Benchmarks added if relevant.
6. **AI Disclosure** – If AI generated, `[AI]` tag present and review performed.
7. **ADR** – If a breaking change, an ADR exists.

---

## 8. Amendments
- The constitution is versioned (`v1`, `v2`, …) and stored in this file. Any amendment must be submitted as a PR, pass the standard review process, and be merged with a **two‑approver** majority.
